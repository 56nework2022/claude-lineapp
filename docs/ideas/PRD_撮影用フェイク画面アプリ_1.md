## 撮影・演劇用 フェイク画面メーカー(仮称)

> **ステータス:** ドラフト v0.2
> **最終更新:** 2026-06-29
> **作成目的:** アイデアの整理と関係者との認識合わせ。未確定の項目は「TBD(要検討)」と記載。

## 1. 背景・課題

映画・ドラマ・演劇などの映像/舞台制作において、登場人物がLINEなどでやり取りする場面を撮影したいニーズがある。しかし実物のアプリでは、会話内容・送信時刻・名前・アイコン・既読状態などを自由にコントロールできず、撮影現場での再現が難しい。

また、撮影では「画面そのもの」だけでなく「画面の動き・遷移」を見せたい場面が多い。たとえば、役者がスマホを操作して相手を探し、通話をかけ、着信画面を経てトークに切り替わる――といった一連の流れを実機で見せたい。これは、スクリーンショット画像を1枚作るだけの方法では実現できない。

本アプリは、**撮影用スマホの実機画面にリアルなフェイク画面(LINE風トークなど)を表示**し、制作者が会話や画面を自由に作り込めるようにすることで、この課題を解決する。

---

## 2. ターゲットユーザー

- **メイン:** 映画・ドラマ・自主制作映画・演劇などの制作スタッフ(小道具担当、美術、監督、撮影部など)
- **サブ(想定):** YouTube・ショート動画などの個人クリエイター、写真作品の制作者

特に**アマチュア映画制作者・学生・コストを抑えたい制作現場**を重点ターゲットとする。利用シーンの特徴として、現場で素早く呼び出して使えること、そして実機画面として本物に見えることが強く求められる。

---

## 3. プロダクトコンセプト

> **「撮影現場で、本物そっくりのスマホ画面を、実機にそのまま表示できるアプリ」**

単なる画像生成ではなく、**実機表示(現場でのライブ運用)**を主目的とする点が最大の特徴。撮影用スマホ1台にアプリを入れて、そのまま役者に持たせて画面を映す、1台完結の運用を前提とする。

---

## 4. 提供価値(なぜ既存手段より良いか)

- 会話内容・時刻・名前・アイコン・既読などを完全に自由に作り込める
- ステータスバー(時刻・電波・電池)まで含めて画面全体が本物に見える
- 撮影中に通知やホームバーが映り込まないフルスクリーン表示ができる
- 「メッセージが届く」「入力中…」「着信」などの動きを現場でタップ進行できる
- 1つの作品(プロジェクト)に複数の画面を持たせ、撮影中に切り替えられる
- 作った画面を画像として書き出し、スタッフへの連絡・資料作りにも使える

---

## 5. アーキテクチャ方針

### 5.1 基本構造:共通インフラ + 種類別エディタ

再現したい画面は構造によって2タイプに分かれるため、無理に1つに統合しない。

- **チャット型:** LINEのトーク、Instagram DM、Messenger など(吹き出しが縦に並ぶ構造。共通化しやすい)
- **フィード型:** Xのタイムライン、Instagramの投稿 など(投稿に「いいね・リポスト・コメント」がぶら下がる別構造)

**共通化する部分(土台のインフラ):**
- 登場人物の管理(名前・アイコン)
- 時刻・ステータスバー(電波・電池)の設定
- 撮影用フルスクリーン表示モード
- 画像の書き出し

**種類ごとに分ける部分:**
- 会話/投稿を組み立てる編集画面そのもの(チャット型・フィード型で別々に作る)

### 5.2 操作をシンプルに保つ仕組み:テンプレート先行方式

新規作成時にまず「何を作るか(LINE / X / Instagram…)」を選ばせ、選んだ種類専用の編集画面だけを表示する。これにより、1画面に全機能が詰め込まれて複雑化するのを防ぐ。ユーザーは常に「自分が選んだ1種類分」の操作だけを見る。

### 5.3 データ階層

```
アプリ(1本)
 └─ プロジェクト(作品)※複数保存可能
     └─ 画面 ※1プロジェクトに複数持てる
         例:トーク画面 / 着信画面 / 通話画面 / 通知 …
```

撮影では1作品で複数シーン分の画面が必要になるため、名前をつけて複数保存でき、現場でパッと呼び出せることを前提とする。

## 6. 主要機能と優先順位(画面の種類)

LINEを例に、撮影での登場頻度と作りやすさで仕分けたフェーズ分け。

### フェーズ1(MVP)
- **トーク画面**(主役。吹き出しのやり取り、時刻・既読・アイコン設定)
- **ロック画面/ホーム画面の通知**(登場頻度が高く、作りも比較的シンプル)
- **画像書き出し**(撮影前のスタッフ共有・資料作りで需要あり)
- 撮影用フルスクリーン表示モード
- 登場人物・ステータスバーの設定
- プロジェクトの複数保存・呼び出し

### フェーズ2
- **着信画面**(相手名+応答/拒否ボタン)
- **通話画面**(音声・ビデオ。芝居で需要が高い)
- **トーク一覧/友だちリスト**
- SNS系スキン(Instagram DM、X投稿などフィード型を含む)

### フェーズ3以降
- プロフィール画面、その他細かい画面
- AIチャット系スキン(ChatGPT・Claude など。構造・演出が異なるため別物として丁寧に作る)

## 7. リリース方針

- **フェーズ1はLINE風スキンのみで単一アプリとしてリリース**し、市場の反応を見る
- 中身は最初から「共通インフラ + 種類別エディタ」で作るため、SNS・AIチャットは将来のスキン/エディタ追加として足せる
- 運用してみて別客層・別ニーズが明確になれば、その時点で別アプリへの切り出しを検討(フィード型やAIチャット型が自然な分割の境界線になりうる)

複数アプリの同時量産は運用コスト増・ストア審査リスクがあるため、初期は避ける。

## 8. 収益化モデル

**基本方針:フリーミアム(核機能は完全無料 + 追加機能は買い切り)**

ターゲットがアマチュア映画制作者・学生・コストを抑えたい層であるため、「お金がない人ほど使ってほしい」という思想を最優先する。高額サブスクや高い初期費用は、最も使ってほしい層を弾くため採用しない。

**無料(透かしなし・撮影に支障なく使える):**
- LINE風トーク画面の作成
- 実機フルスクリーン表示
- ロック画面/通知
- 画像書き出し
- 登場人物・ステータスバー設定
- プロジェクトの複数保存
- ※ フェーズ1(MVP)一式を無料とし、「お金がない人もLINE画面の撮影には不自由しない」を担保する

**買い切り(余裕のある人・もっと欲しい人向け):**
- 追加スキン(SNS系、AIチャット系)
- 着信・通話画面
- 機種の細かい作り分け など
- ※ フェーズ2以降の、作り込みが重い機能を有料とする
- 価格帯: TBD

**ウォーターマーク(透かし)の方針:**
- 実機表示・画像書き出しともに無料でも透かしなし(撮影に映り込む・素材として使えなくなるため)

## 9. 制約・前提

- **知的財産権:** LINE・Instagram・X などは実在サービス。ロゴや正確な名称をそのまま使うと商標・著作権上の問題が生じうる。「〜風」にとどめ、公式ロゴ・正確な名称の使用は避ける方針とする。(要法務確認)
- **対応OS:** iOS / Android の両対応。なお、ステータスバー(時刻・電池・電波の位置)、通知の出方、着信・通話画面のデザインは iPhone と Android で見た目が異なるため、各スキン内で「iPhone表示 / Android表示」を切り替えられる構造とする。撮影では機種の見た目が画に映るため重要。
- **撮影実機への表示方法:** 撮影用スマホ1台にアプリを入れ、そのまま画面を見せる1台完結の運用を前提とする。
- **ユーザー登録・アカウント機能:** 不要。データは端末ローカル保存のため技術的な必要がなく、「ダウンロード→即起動→すぐ使える」体験を最優先する。課金管理はApp Store / Google Playに委ねる。将来的に複数端末同期やチーム共有のニーズが出た場合に検討する。
- **開発体制:** ひとり開発。スケジュール・予算は現時点では未定。機能の優先順位付けとフェーズ分けはこの制約を前提に組んでいる。

## 10. 成功指標(KPI)

優先順位順:

1. **ダウンロード数** — とにかく使ってもらえたか。「お金がない人に使ってほしい」という方針と直結する最重要指標
2. **有料転換率** — 買い切りに課金してくれる人の割合。持続可能性の指標
3. **継続利用率** — 撮影のたびに使い続けてくれるか。本当に現場で役立っているかの証左
4. **レビュー・口コミの質** — 「現場で本当に役立ったか」の定性指標。改善方針の判断材料にもなる

具体的な数値目標は、リリース後の状況を見て設定する。

# 撮影・演劇用フェイク画面メーカー アプリ開発

## プロジェクト概要

映画・ドラマ・演劇などの撮影現場で、スマホの実機画面にリアルなLINE風のフェイク画面を表示するFlutterアプリを開発する。スクリーンショット生成ツールではなく、「撮影用スマホ1台に入れて、そのまま役者に持たせて画面を映す」実機表示ツールが主目的。

## 技術スタック

- Flutter / Dart(iOS・Android 両対応)
- データ保存: 端末ローカルのみ(hive を使用)
- 外部サービス・クラウド: 不使用
- 状態管理: Riverpod

## アーキテクチャ方針

**共通インフラ + 種類別エディタ**の構造で作る。

- 共通インフラ: 登場人物管理・ステータスバー設定・フルスクリーン表示・画像書き出し・プロジェクト保存
- 種類別エディタ: チャット型(LINE風)・フィード型(SNS風)を別々のウィジェットで実装。フェーズ1はチャット型のみ
- 新規作成時に「何を作るか」を先に選ばせるテンプレート先行方式。ユーザーは選んだ種類の編集画面だけを見る

**データ階層:**
```
プロジェクト(Project)
 └─ 画面(Scene) ※複数持てる
     └─ 種類(type: chat / feed)
     └─ 登場人物リスト(Character[])
     └─ メッセージリスト(Message[])
     └─ ステータスバー設定(StatusBarConfig)
```

## フェーズ1(MVP)のスコープ ← まずここだけ作る

以下のみ実装する。それ以外は後回し。

1. **プロジェクト一覧画面** — プロジェクトの作成・一覧表示・削除。hiveでローカル保存
2. **画面一覧画面** — プロジェクト内の画面(Scene)を複数管理。新規作成時に種類を選ぶ(フェーズ1はLINE風のみ)
3. **登場人物エディタ** — 名前・アイコン画像を設定。最低2人(送信側・受信側)
4. **ステータスバー設定** — 時刻(任意入力 or 現在時刻)・電波・電池・iPhone/Android切り替え
5. **LINE風トークエディタ** — 吹き出しを追加・編集・並び替え。送信者・本文・時刻・既読の設定
6. **撮影用フルスクリーン表示モード** — システムUI(ステータスバー・ホームバー・通知)を完全に隠し、アプリ描画で上書き。タップで次のメッセージを「届かせる」アニメーションを進行できる
7. **ロック画面/通知プレビュー** — 相手名・メッセージ冒頭・時刻を表示するシンプルな通知風画面
8. **画像書き出し** — 現在表示中の画面をPNG画像としてカメラロールに保存

## 重要な設計上の注意点

- **フルスクリーン表示が最重要**: `SystemChrome.setEnabledSystemUIMode(SystemUiMode.immersiveSticky)` を使い、OS のステータスバーを完全に隠して、アプリ側で描画したステータスバーで上書きする。これが本物らしさの核になる
- **透かし(ウォーターマーク)は一切入れない**: 無料版でも透かしなし。撮影に映り込むと小道具として使えなくなる
- **iPhone/Android表示の切り替え**: ステータスバーのレイアウト(時刻位置・ノッチ・電池アイコン)をiPhone風/Android風で切り替えられるウィジェットを共通化する
- **既存サービスのロゴ・名称は使わない**: 「LINE風」にとどめ、LINEのロゴや正式名称は使用しない

## 最初にやってほしいこと

1. Flutterプロジェクトを作成し、hive・Riverpod・`screenshot`パッケージ(画像書き出し用)を導入する
2. 上記データ階層のモデルクラス(Project / Scene / Character / Message / StatusBarConfig)を作成する
3. プロジェクト一覧画面から始めて、画面一覧 → エディタ → フルスクリーン表示の順で実装を進める
4. 各ステップで実機(またはシミュレータ)で動作確認しながら進めること
